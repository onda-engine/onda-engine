# Premium Composition — Deep Research (what truly reads premium + how to build it)
_Verified web research (Apple WWDC, Stripe deconstructions, VFX/grading sources). Companion to premium-composition-gaps.md._

## Thesis
What separates genuinely premium brand films from templated motion graphics is not polish but COMMITMENT TO A CONCEPT plus a stack of "invisible" craft passes—color grading that unifies every element into one look, naturalistic motion blur (the 180-degree shutter), light-wrap and grain-match passes that make composited elements share one camera's fingerprint, and FBM-noise gradient structure rather than smooth blobs. A 2D vector + GPU engine can genuinely reach the Stripe/Linear/Vercel tier in pure vector (live WebGL/WebGPU FBM gradients, light sweeps, grade, grain, kinetic type), and can reach CINEMATIC premium by treating AI-generated media as the photoreal ingredient and the engine as the DIRECTOR/COMPOSITOR—applying the exact VFX unification workflow (analyze, light-match, track, composite/matchmove, then grade+grain+light-wrap in linear space) used to integrate CG into live-action. The hard, non-negotiable technical disciplines are: composite in LINEAR light with the output transform/LUT applied last (a log-look LUT on linear data breaks), grade in three luminance regions (shadows/mids/highlights, not global) with a subtle warm-midtone push, and add both a grain-match and a light-wrap pass so AI media doesn't read as "pasted on." For live in-browser heroes, premium sites like Stripe DO run real-time shader renders (live <canvas>, minigl, FBM over Simplex noise at ~60fps), achievable jank-free with correct frame pacing—triple-buffer on Android (2 is fine on desktop), tune max frame latency to 1–3, and present via buffer-pointer swap during vsync rather than readback. The honest ceiling: pure flat vector reaches the Linear/Stripe gradient-and-glass aesthetic, but true Apple-product-film photorealism fundamentally requires either 3D or composited photoreal media—so the right north star is "compositor/director over AI-generated photoreal," not a hand-rolled 3D render farm.

## Findings

### [medium · 2-1] Concept/storytelling commitment — a single cohesive idea reinforcing the brand — is the PRIMARY differentiator of premium motion design, not polish. Templated work is 'utilitarian' with no concept tied to the branding; premium work has a visual language and storytelling that aligns with brand identity.
School of Motion's $7-vs-$1000 breakdown: the cheap work was 'incredibly utilitarian' with 'no concept or custom design that reinforced the branding,' while the pro work had 'visual language and storytelling' aligned to brand identity. Independently reaffirmed by 2025-26 sources (Envato Elements: 'Intentionality Over Polish... creative direction, strategy, and taste are more valuable than ever'; theTeam, Dusted, Maerco). Caveat: the anecdote is n=1, figures are dated, and the source author sells courses (mild bias); the article also lists concept alongside animation quality and sound design, so 'primary' slightly over-weights one of several factors. Practical implication for ONDA: art-direction DEFAULTS and a single committed metaphor matter more than adding more components.

### [high · 3-0] Premium 'expensive-looking' gradients come from layering multiple octaves of Simplex noise via Fractal Brownian Motion (FBM) — higher frequency (lacunarity) and lower amplitude (persistence) per octave — producing 'wispy' detailed structure rather than smooth blobs. This is exactly the technique behind Stripe/Linear-tier gradients.
Primary repo: 'This technique layers multiple octaves of Simplex noise, where each successive layer has a higher frequency (lacunarity) and lower amplitude (persistence)... creates the wispy and detailed structures... rather than just smooth blobs.' Corroborated by Book of Shaders ch.13 (canonical), FastNoise2, PBR-Book, Inigo Quilez fBM: single-octave noise = smooth/low-detail; FBM = progressive fine detail. Decades-stable signal-processing fact (low-freq octave = broad shape, high-freq octaves = fine detail). Directly implementable in a GPU fragment shader; no 3D geometry needed.

### [high · 3-0] Stripe's signature animated hero gradient is a REAL-TIME live WebGL render (drawn to <canvas id="gradient-canvas"> via a minimal 'minigl' + a Gradient class running GLSL FBM over 3-octave Simplex noise at ~60fps), NOT a pre-rendered video loop. The diagonal edge is a CSS skewY(-12deg)+overflow:hidden trick over a full-rectangle live canvas. This proves a flat-2D vector/GPU engine can deliver premium live heroes.
Variable names (minigl, Gradient class) lifted from Stripe's deployed minified JS — not speculation. Multiple INDEPENDENT reverse-engineers (bram.us/Bramus Van Damme, Alex Harri full deconstruction, dkaraush shader gist, wave-gradient/gradient-stripe repos) reach the identical conclusion: live <canvas> running GLSL fragment shaders (FBM over 3-octave Simplex noise, time-modulated UVs) at runtime ~60fps, explicitly NOT a video loop. Verifiable by anyone via DevTools on stripe.com. Architecture stable for years; no contradicting source. (Refuted sub-claim: kevinhufnagl alone is incomplete on the exact warp mechanism — rely on Alex Harri / dkaraush for the shader internals.)

### [high · 3-0] The 180-degree shutter angle is the cinematic default that produces naturalistic motion blur; shutter speed = 1/(2×fps), so at 23.98fps it equals 1/47.95 sec. The general formula is Shutter Speed = 1/(Frame Rate × 360/Angle), so a fixed angle gives a frame-rate-dependent exposure (the basis of the '180-degree rule'). This is the concrete numeric default for 'expensive' motion blur in a render engine.
RED primary docs state verbatim 'The default shutter angle is 180°' and 'The default shutter speed is 1/47.95 sec,' with the conversion table and formula 'Shutter Speed = 1/(Frame Rate × 360/Angle)' plus the inverse. Math checks: 1/(2×23.98)=1/47.96≈1/47.95; 180° at 24fps = 1/48s. Decades-stable standard inherited from rotary-shutter film cameras (open half the frame interval = half exposure). Corroborated by StudioBinder, ProVideo Coalition, In Depth Cine. For ONDA's motion-blur pass: sample/accumulate over 50% of the frame interval (180°) as the default to read cinematic rather than strobing-sharp or smeary.

### [high · 3-0] Professional VFX integration of a foreground element (live-action or 3D CG) over a background plate follows a five-step workflow: (1) analyze the shot, (2) light the element to match, (3) motion-track the background plate, (4) composite/matchmove foreground over background, (5) fine-tune color correct adding grading, grain, atmospherics, artifacts. This is the exact pipeline a vector engine should run when compositing AI-generated media.
O'Reilly 'Digital Visual Effects' ch.5 lists exactly these five numbered steps verbatim. Corroborated by CG Spectrum, LucidLink (2026 pipeline guide), MASV: canonical track → light/render → composite → match color/contrast/grain/atmospherics. Caveat: 'fixed' is mildly strong — real pipelines aren't strictly linear (tracking often precedes lighting; virtual production reorders) — but the five activities and their substance are settled. Maps directly to ONDA's compositing-of-AI-media flow.

### [high · 3-0] The core task of premium compositing is making multiple image sources (plate + synthetic/AI renders) look like ONE camera captured them in ONE moment — matching color temperature, exposure, black levels, and contrast so 'nothing looks added.' Light wraps and grain/noise matching are two DISTINCT named procedural passes both required to unify a composite; AI-generated clips carry color variations that demand the identical grade-to-match + grain-match + light-wrap treatment used for CG-to-plate.
cglounge: 'combining multiple image elements into a single frame that looks like it was captured by one camera in one moment... matching color temperature... Black levels, color temperature, exposure, highlight response. All of it has to match.' O'Reilly ch.5 names both 'Matching Film Grain and Video Noise' and 'Creating Procedural Light Wraps' as distinct passes. Corroborated by Boris FX, Foundry/Nuke docs (ScannedGrain node, light-wrap tutorials), Rebelway. 2024-26 sources confirm the same passes unify AI footage: 'light wrapping blends edges and matches color tones... adding grain helps AI-generated elements blend seamlessly.' Caveat: 'needs/exact' is slightly strong — a deliberately flat/graphic motion-graphics frame need not make AI media invisible the way seamless CG-to-plate must — but the unification problem (match grade/grain/light so media doesn't read pasted-in) genuinely transfers. (Refuted over-strong variants: 'lighting match is THE single most important factor' 0-3; 'grain match is non-negotiable' 0-3; lens-imperfection recipe 1-2 — treat these as helpful, not mandatory.)

### [high · 3-0] Compositing must be done in LINEAR light: convert the plate to linear, do ALL blending/compositing in linear space, and apply the output transform/LUT only at the very END. Applying a log-to-look LUT to a linear render produces broken results because the LUT's input was calibrated for log, not linear, code values. This is a non-negotiable technical discipline for an engine that composites AI media.
Settled VFX color-management consensus (linear workflow / ACES / OCIO). Chris Brejon: 'Just like rendering, compositing should be done in linear' — pixel ops (motion blur, defocus, resizing) are 'more physically plausible' and light mixing 'preserves the appearance.' Prolost/Stu Maschwitz: linear-light ops 'look better, more organic, and more realistic.' Log-LUT-on-linear breakage confirmed by ARRI guidance + CreativeCOW: log-encoded LUTs 'anticipate logarithmic input, not linear input'; a LUT is indexed by input code values in a specific encoding, so feeding the wrong encoding gives wrong output. Foundational, not fast-moving. ONDA implication: blend/grade in linear, tonemap/output-transform last.

### [high · 3-0] To match CG/AI media to live-action, grade in THREE separate luminance regions (shadows/lifted blacks, midtones, highlights) rather than one global adjustment, with a subtle warm-midtone push that raises red and green channels slightly relative to blue (because daylight CG/AI renders cooler than warm sun-lit plates).
Source: 'Don't try to fix everything with a single overall brightness adjustment—that never works. Work in three passes,' with a 'subtle warm push in the midtones (raising the red and green channels slightly relative to blue).' Three-way correction (lift/gamma/gain = shadows/mids/highlights) is the foundation of every grading tool; DaVinci Resolve docs describe lift/gamma/gain as overlapping tonal ranges with log wheels 'to let you adjust one tonal area without affecting others.' Properly hedged ('subtle,' 'slightly'). Physically grounded. Directly implementable as a 3-region grade node in ONDA's pipeline.

### [high · 3-0] Premium hero loops CAN be live in-browser real-time renders, and jank-free 60fps presentation requires concrete buffering/pacing: minimum swapchain size 3 on Android (2 is fine on desktop), maximum frame latency tuned to 1–3 frame periods (2 for most cases; 1 if sim+render fits one frame; 3 to prioritize avoiding drops over latency), and buffer swap by repointing front/back pointers (not copying pixels) timed to the vertical-sync window.
Raph Levien (Vello/piet-gpu author — the very renderer ONDA uses): 'minimum acceptable swapchain size to sustain a smooth framerate is 3 for Android, while 2 is fine on desktop'; 'A value of 2 might make sense for most games, but 1 might be better if the entire loop of simulation + rendering can be accomplished in a single frame, and 3 if there's more concern about frame drops than latency.' Vulkan-Samples + Arm/Samsung confirm under FIFO, double-buffering collapses to 30fps on any frame miss while triple-buffering keeps 60fps (Mali G72 benchmark). Microsoft DXGI confirms latency 3 default / 2 for CPU-GPU parallelism / 1 for least latency. Carmen Cincotti: 'the GPU adapter changes its frontbuffer pointer to the backbuffer and vice-versa' during the vsync window — pointer swap, not pixel copy. (Refuted: the specific 'present path uses double buffering / readback flickers' framings split 1-2 — present via swapchain pointer-swap, and avoid CPU readback for a live hero.)

### [medium · 2-1] AI video tools (Runway) support an in-tool compositing/finishing workflow — masking/green-screen with alpha export, relight, color-grade (LUT export), layer compositing — and produce compositing-ready files (ProRes 4444 with alpha) that drop into After Effects/Premiere, making AI media a practical photoreal INGREDIENT to be directed and composited rather than a finished output. This validates the 'generate the photoreal, then direct + composite it' workflow for a vector engine.
Runway ships a dedicated 'Relight Scene' app, Aleph (relight, background removal, object replace, reframe), Green Screen with alpha-matte export, text-to-color-grade outputting .cube LUTs, Motion Brush, and node-based Workflows; Gen-4 exports ProRes 4444 with alpha 'for easy compositing in tools like After Effects.' Caveat (the 1 dissent): 'export directly to AE/Premiere' overstates integration — there is no native push plugin yet (an Adobe–Runway partnership is rolling out); the real flow is exporting ProRes/alpha/LUT and importing. The load-bearing point holds: AI footage is finishable and compositing-ready. (Refuted: the specific 'Sora 2 to prototype then Runway to refine' hybrid framing split 1-2 — treat tool choice as fluid; the principle is generate-then-composite, not a fixed two-tool chain.)
